# Roadmap: echart-skill v2.0 → Agent BI System

**Created:** 2026-06-13
**Status:** Planning
**Goal:** 从"图表生成工具"升级为"AI 数据分析助手"——不止画图，更能理解数据、发现问题、生成洞察、撰写报告。

---

## 一、竞品深度分析

### 1.1 对标项目总览

| 项目 | Stars | 核心定位 | 关键能力 | 与我们最相关的点 |
|------|-------|---------|---------|----------------|
| **WrenAI** | 5.3k+ | GenBI 平台 | 语义模型(MDL) + Text-to-SQL + RAG | 语义层设计 |
| **DeepAnalyze** | 1.5k+ | AI 数据分析师 | 端到端自动分析 + 报告生成 | 洞察引擎 + 报告 |
| **LAMBDA** | - | 无代码分析 | Programmer+Inspector 双 Agent 协作 | 多 Agent 架构 |
| **OpenChatBI** | 新 | 对话式 BI | 会话记忆 + 异常检测 + 预测 + MCP | 对话交互 + 高级分析 |
| **FinSight** | 新 | 金融深度研究 | 4 Agent 流水线 + 2 万字报告 | 专业报告生成 |
| **Lumen AI** | - | Agent 框架 | 声明式数据模型 + RAG + Pipeline | 可扩展架构 |
| **Multi-Agent BA** | 新 | 离线业务分析 | 5 角色 Agent + 完全离线 | 隐私优先多 Agent |

### 1.2 关键模式总结

**模式 1：多 Agent 协作架构**
```
数据采集 Agent → 分析 Agent → 校验 Agent → 报告 Agent
```
几乎所有头部项目都采用多 Agent 分工，而不是单一模型做所有事。

**模式 2：语义层是核心壁垒**
WrenAI 最成功的点是 MDL（Modeling Definition Language）——让 AI 理解"净收入"对应哪个 SQL 表达式。这不是 Vector Search 能解决的，需要显式建模。

**模式 3：输出不止于图表**
DeepAnalyze 和 FinSight 证明：用户最终要的不是图表，是"结论"。图表是结论的佐证，不是目的。

**模式 4：上下文记忆是关键体验**
OpenChatBI 的对话记忆让用户能像和人对话一样追问数据。没有记忆的分析工具每次都是"从零开始"。

### 1.3 我们 vs. 竞品：差异定位

| 维度 | 竞品 | echart-skill 优势 |
|------|------|------------------|
| 部署复杂度 | 多数需要 Docker/服务 | 我们的纯本地 Python 脚本，零配置 |
| 数据隐私 | 多数需要 LLM API | 我们可以做本地模型（Ollama）+ DuckDB 完全离线 |
| 图表质量 | 多数用简单的 matplotlib/vega | 我们的 ECharts 6.0 + 356 案例 + 41 模板是绝对优势 |
| 平台集成 | Web UI 为主 | 我们的 Agent Skill 模式天然支持 Claude Code 等 AI 编程助手 |
| 中文支持 | 多数英文优先 | 我们在中文场景（地图、表头、指令）有深度优化 |

**定位**：做最好的**本地优先、隐私安全、中文友好**的 Agent BI 系统。用 Agent Skill 形态嵌入各类 AI 编程助手，让用户在自己的 IDE/终端里就能完成专业数据分析。

### 2.2 核心模块设计

#### 模块 1：洞察引擎 (Insight Engine)

```
Input: 表名 + 分析维度
  │
  ├── 1. 数据画像 (Data Profiling)
  │     - 列类型、缺失率、唯一值数、分布特征
  │     - 自动识别：时间列、类别列、数值列、地理列
  │
  ├── 2. 多维聚合 (Multi-dim Aggregation)
  │     - 按时间粒度聚合（日/周/月/季/年）
  │     - 按类别维度聚合
  │     - 交叉维度组合
  │
  ├── 3. 模式发现 (Pattern Discovery)
  │     - 趋势检测：上升/下降/平稳
  │     - 异常检测：Z-score / IQR / 同比偏离
  │     - 拐点检测：变化率突变
  │     - 相关性：数值列之间的 Pearson/Spearman
  │     - 周期性：按周/月的规律波动
  │
  ├── 4. 洞察生成 (Insight Generation)
  │     - Top-N 发现："浙江、广东、江苏贡献了 62% 的销售额"
  │     - 变化描述："本月环比下降 12%，主要受浙江省拖累"
  │     - 异常告警："3月15日订单量异常飙升 340%"
  │     - 对比分析："Q2 vs Q1 同比增长 18%，增速加快"
  │
  └── 5. 洞察排序 (Insight Ranking)
        - 按 Impact 大小排序
        - 按置信度过滤
        - 去重合并相似洞察

Output: 结构化的洞察列表 + 自然语言描述
```

#### 模块 2：报告引擎 (Report Engine)

```
Input: 分析主题 + 洞察列表 + 图表
  │
  ├── 1. 报告结构规划
  │     - 摘要 (Executive Summary)
  │     - 关键指标仪表盘 (KPI Cards)
  │     - 核心发现 (Key Findings)
  │     - 详细分析 (Detailed Analysis)
  │     - 建议与下一步 (Recommendations)
  │
  ├── 2. 模板系统
  │     - 通用数据分析报告模板
  │     - 销售分析报告模板
  │     - 财务分析报告模板
  │     - 运营分析报告模板
  │     - 用户自定义模板
  │
  ├── 3. 内容生成
  │     - Markdown 格式（基础）
  │     - PDF 格式（WeasyPrint/pandoc）
  │     - HTML 格式（交互式）
  │
  └── 4. 图表嵌入
        - 将生成的 ECharts 图表嵌入报告
        - 支持静态截图(PNG)或交互式(HTML)
```

#### 模块 3：语义层 (Semantic Layer)

```
语义模型定义（升级 metrics 模块）:
{
  "business_models": {
    "销售订单": {
      "table": "sales_orders",
      "description": "所有销售订单记录",
      "columns": {
        "order_id": {"type": "id", "description": "订单编号"},
        "order_date": {"type": "date", "description": "下单日期", "granularity": "day"},
        "customer_name": {"type": "category", "description": "客户名称"},
        "region": {"type": "geo", "description": "所属区域", "geo_level": "province"},
        "amount": {"type": "metric", "description": "订单金额(元)", "unit": "元", "agg_default": "sum"},
        "quantity": {"type": "metric", "description": "订购数量", "agg_default": "sum"},
        "category": {"type": "category", "description": "产品类别"}
      },
      "metrics": {
        "总销售额": "SUM(amount)",
        "平均客单价": "AVG(amount)",
        "订单数": "COUNT(DISTINCT order_id)",
        "客均商品数": "AVG(quantity)"
      },
      "relations": {
        "products": {"type": "LEFT JOIN", "on": "sales_orders.product_id = products.id"},
        "customers": {"type": "LEFT JOIN", "on": "sales_orders.customer_id = customers.id"}
      }
    }
  }
}
```

#### 模块 4：对话管理 (Context Manager)

```
会话状态:
  - 当前分析表
  - 最近查询结果引用
  - 分析焦点（维度、指标、时间范围）
  - 历史图表引用
  - 追问链："上个月" → "和去年同期比" → "为什么"

实现方案:
  - 轻量级：JSON 文件存储会话状态
  - 本地 SQLite 管理对话历史
  - 支持 "撤销上一步" 的天然 undo 机制
```

## 四、技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 洞察生成方式 | 规则引擎 + LLM 总结 | 规则保证准确，LLM 让表达自然 |
| 报告格式 | Markdown → HTML → PDF | 渐进增强，MD 最简单，HTML 可交互 |
| 语义层存储 | JSON/YAML 文件 + DuckDB 元数据表 | 轻量、可版本控制、Git 友好 |
| 对话记忆 | SQLite 轻量存储 | 与现有 DuckDB 一致性，单文件部署 |
| 预测模型 | Python statsmodels + 手写简单方法 | 避免重依赖，DuckDB 也有基础统计函数 |
| 本地模型支持 | Ollama 可选接入 | 保持零配置默认体验，高级用户可选 |

## 五、成功指标

| 指标 | 当前(v1.5) | 目标(v2.0) |
|------|-----------|-----------|
| 用户分析流程 | 手动写 SQL → 选图表 → 看结果 | 一句话描述 → 自动分析 → 拿到结论 |
| 输出形式 | 单一 HTML 图表/Dashboard | 图表 + 洞察文字 + 分析报告 |
| 分析深度 | 单次查询、单维度 | 多维度交叉、趋势+异常+对比 |
| 可追问性 | 不支持 | 支持自然追问和上下文记忆 |
| 数据理解 | Agent 只知道列名和类型 | Agent 理解业务含义和指标定义 |
| 报告能力 | 无 | 一键生成专业 Markdown/PDF 报告 |
